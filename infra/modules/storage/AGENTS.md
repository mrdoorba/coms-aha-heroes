# storage

## Purpose
Provisions two GCS buckets for application file storage: an exports bucket with a 90-day lifecycle delete policy, and an uploads bucket with CORS configuration for direct browser uploads from Cloud Run origins.

## Key Files
| File | Description |
|------|-------------|
| `main.tf` | Creates `coms-aha-heroes-exports` (90-day lifecycle delete rule, uniform bucket-level access) and `coms-aha-heroes-uploads` (no lifecycle/permanent retention, CORS allowing GET/PUT/POST from `*.run.app`, uniform bucket-level access) |
| `variables.tf` | Input variables: `project_id`, `region` — note: `region` default is `ASIA-SOUTHEAST2` (uppercase), which is overridden by the root module passing the lowercase value |
| `outputs.tf` | Exposes bucket names for use by `cloud-run/` IAM bindings |

## For AI Agents

### Working In This Directory
- The `region` variable default is uppercase (`ASIA-SOUTHEAST2`) but GCS accepts both forms. The root module overrides this with the lowercase `asia-southeast2` value — do not rely on the default in new code.
- CORS on `coms-aha-heroes-uploads` allows `*.run.app` origins. If the app moves to a custom domain, add that origin to the CORS config in `main.tf`.
- `coms-aha-heroes-uploads` has **no lifecycle delete rule** — files accumulate permanently. Implement application-level cleanup if storage costs grow.
- Both buckets use uniform bucket-level access (no per-object ACLs). IAM on the runtime SA (`cloud-run/`) grants object-level permissions.
- V4 signed URLs (generated by Cloud Run using `serviceAccountTokenCreator`) are used for direct browser uploads to avoid proxying large files through the app server.

### Common Patterns
- Upload flow: browser requests signed URL from app → app generates V4 signed URL → browser uploads directly to GCS.
- Export flow: app writes to exports bucket → 90-day lifecycle ensures automatic cleanup of old reports.
- Bucket name format: `coms-aha-heroes-{purpose}` — keep this consistent if adding new buckets.
