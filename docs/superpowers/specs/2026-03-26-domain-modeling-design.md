## 1. Overview

Translate the SQL DDL from `docs/architecture.md` Section 6 into a type-safe TypeScript domain layer using Drizzle ORM, Zod, and shared constants. Also update the architecture doc's monorepo structure and references to reflect the TanStack Start v1 migration.

### What this covers

- **Drizzle ORM schema** — 7 schema files grouped by bounded context
- **Zod validation schemas** — request input and query param validation for all API endpoints
- **TypeScript types** — inferred DB types + derived response types
- **Constants/enums** — roles, statuses, KITTA categories, notification types, settings keys
- **Architecture doc update** — Section 5 monorepo structure + scattered Next.js references

### What this does NOT cover

- Repository layer (`src/server/repositories/`) — designed with API routes
- Service layer (`src/server/services/`) — designed with business logic
- Migration files — generated by `drizzle-kit generate` after schema is written
- Seed data — separate task after migrations exist
- DB triggers/functions (updated_at, point_summary sync) — raw SQL in migration seed

### Bounded context grouping rationale

| Context | Tables | Why grouped |
|---------|--------|-------------|
| branches | branches, system_settings | Tenant config — rarely changes, referenced by everything |
| users | users, teams | Identity + org structure, circular FK (team.leader_id → users) |
| points | achievement_points, point_categories, point_category_translations, point_summaries | Core transaction + its lookup tables and materialized view |
| disputes | challenges, appeals | Both are dispute mechanisms on achievement_points, separate auth rules |
| rewards | rewards, redemptions | Catalog + spend transactions |
| social | comments, notifications | User-facing communication, polymorphic patterns |
| audit | audit_logs, sheet_sync_jobs | Immutable records, admin-only |

## 3. Drizzle Schema Details

### 3.1 Common patterns

All schema files follow these conventions:

- **UUID primary keys** — `uuid('id').primaryKey().defaultRandom()`
- **Timestamps** — `timestamp('created_at', { withTimezone: true, mode: 'date' }).notNull().defaultNow()`
- **Foreign keys** — inline `.references(() => table.id)` for simple FKs
- **Enums** — `pgEnum` for PostgreSQL enums (`user_role`, `point_status`, `redemption_status`)
- **CHECK constraints** — Drizzle `check()` for value restrictions (kitta_component, point_categories.code)
- **Indexes** — `index('idx_name').on(table.column)` matching the architecture SQL indexes
- **Relations** — defined via `relations()` for Drizzle's relational query API

### 3.2 `branches.ts`

```typescript
// Tables: branches, systemSettings
// Enums: none
// Relations: systemSettings.updatedBy → users (defined in users.ts to avoid circular import)
```

`branches` — id, code (varchar 10, unique), name (varchar 100), timezone (varchar 50, default 'Asia/Jakarta'), locale (varchar 10, default 'id'), created_at, updated_at.

`systemSettings` — key (varchar 100, PK), value (jsonb), description (text), updated_by (uuid FK → users), updated_at. No `created_at` — settings are seeded, not user-created.

### 3.3 `users.ts`

```typescript
// Tables: users, teams
// Enums: userRoleEnum
// Relations: users ↔ teams (bidirectional), teams.leader → users
```

`userRoleEnum` — `pgEnum('user_role', ['admin', 'hr', 'leader', 'employee'])`

`teams` — id, branch_id FK, name (varchar 100), leader_id (uuid, nullable — circular FK resolved via separate `.references()`), created_at, updated_at. Indexes: branch_id, leader_id.

`users` — id, branch_id FK, team_id FK (nullable), email (varchar 255, unique), name (varchar 255), employee_id (varchar 50, nullable), role (userRoleEnum, default 'employee'), department (varchar 100, nullable), position (varchar 100, nullable), avatar_url (text, nullable), locale_pref (varchar 10, nullable), must_change_password (boolean, default true), is_active (boolean, default true), archived_at (timestamp, nullable), created_at, updated_at. Indexes: branch_id, team_id, email.

Relations:
- `users.branch` → one branches
- `users.team` → one teams (nullable)
- `teams.branch` → one branches
- `teams.leader` → one users (nullable)
- `teams.members` → many users

### 3.4 `points.ts`

```typescript
// Tables: pointCategories, pointCategoryTranslations, achievementPoints, pointSummaries
// Enums: pointStatusEnum
// Relations: achievementPoints → user, submittedBy, approvedBy, revokedBy, category, branch
```

`pointStatusEnum` — `pgEnum('point_status', ['pending', 'active', 'challenged', 'frozen', 'revoked', 'rejected'])`

`pointCategories` — id, code (varchar 50, unique, CHECK IN ('BINTANG', 'PENALTI', 'POIN_AHA')), default_name (varchar 100), description (text, nullable), icon (varchar 50, nullable), requires_screenshot (boolean, default true), is_active (boolean, default true), created_at.

`pointCategoryTranslations` — id, category_id FK, locale (varchar 10), name (varchar 100), description (text, nullable). UNIQUE(category_id, locale).

`achievementPoints` — id, branch_id FK, user_id FK, category_id FK, points (integer, CHECK > 0), reason (text), related_staff (text, nullable), screenshot_url (text, nullable), kitta_component (varchar 2, nullable, CHECK IN ('K', 'I', 'T1', 'T2', 'A') OR NULL), status (pointStatusEnum, default 'pending'), submitted_by FK, approved_by FK (nullable), approved_at (nullable), revoked_by FK (nullable), revoked_at (nullable), revoke_reason (text, nullable), metadata (jsonb, default '{}'), created_at, updated_at. Indexes: branch_id, user_id, status, category_id, submitted_by, created_at.

`pointSummaries` — id, branch_id FK (NOT NULL), user_id FK (NOT NULL, UNIQUE), bintang_count (integer, NOT NULL, default 0), penalti_points_sum (integer, NOT NULL, default 0), direct_poin_aha (integer, NOT NULL, default 0), redeemed_total (integer, NOT NULL, default 0), updated_at. Indexes: (branch_id, bintang_count DESC), (branch_id, direct_poin_aha DESC).

### 3.5 `disputes.ts`

```typescript
// Tables: challenges, appeals
// Enums: none (status is varchar CHECK)
// Relations: → achievementPoints, challenger/appellant → users, resolvedBy → users
```

Both tables share the same shape: id, branch_id FK, achievement_id FK, [challenger_id | appellant_id] FK, reason (text), status (varchar 20, CHECK IN ('open', 'upheld', 'overturned'), default 'open'), resolved_by FK (nullable), resolved_at (nullable), resolution_note (text, nullable), created_at, updated_at. Indexes: achievement_id, branch_id.

### 3.6 `rewards.ts`

```typescript
// Tables: rewards, redemptions
// Enums: redemptionStatusEnum
```

`redemptionStatusEnum` — `pgEnum('redemption_status', ['pending', 'approved', 'rejected'])`

`rewards` — id, branch_id FK (nullable — NULL = global), name (varchar 200), description (text, nullable), point_cost (integer, CHECK > 0), image_url (text, nullable), is_active (boolean, default true), created_at, updated_at. Index: branch_id.

`redemptions` — id, branch_id FK, user_id FK, reward_id FK, points_spent (integer, CHECK > 0), notes (text, nullable), status (redemptionStatusEnum, default 'pending'), approved_by FK (nullable), approved_at (nullable), rejection_reason (text, nullable), created_at, updated_at. Indexes: user_id, branch_id.

### 3.7 `social.ts`

```typescript
// Tables: comments, notifications
// Relations: comments.author → users, notifications.user → users
```

`comments` — id, branch_id FK, entity_type (varchar 30, CHECK IN ('achievement', 'challenge', 'appeal')), entity_id (uuid — polymorphic, no DB FK), author_id FK, body (text), created_at, updated_at. Index: (entity_type, entity_id), branch_id.

`notifications` — id, branch_id FK, user_id FK, type (varchar 50), title (text), body (text, nullable), entity_type (varchar 30, nullable — values: 'achievement', 'challenge', 'appeal', 'redemption'), entity_id (uuid, nullable), is_read (boolean, default false), read_at (timestamp, nullable), created_at. Indexes: (user_id, is_read), (read_at WHERE is_read = true), branch_id. Note: no CHECK constraint on entity_type (unlike comments) because notifications can reference redemptions.

### 3.8 `audit.ts`

```typescript
// Tables: auditLogs, sheetSyncJobs
// Relations: auditLogs.actor → users
```

`auditLogs` — id, branch_id FK, actor_id FK, action (varchar 50), entity_type (varchar 50), entity_id (uuid), old_value (jsonb, nullable), new_value (jsonb, nullable), ip_address (custom inet type via `customType`), created_at. Indexes: branch_id, actor_id, (entity_type, entity_id). No updated_at — immutable.

`sheetSyncJobs` — id, branch_id FK, direction (varchar 10, CHECK IN ('import', 'export')), sheet_id (varchar 255), sheet_name (varchar 100, nullable), status (varchar 20, default 'pending'), rows_processed (integer, default 0), rows_failed (integer, default 0), error_log (jsonb, nullable), started_by FK (nullable), started_at (nullable), completed_at (nullable), created_at.

### 3.9 `index.ts` (barrel)

Re-exports all tables, enums, and relations from all schema files. Single import point:
```typescript
import { users, teams, achievementPoints, ... } from '@/db/schema'
```

### 3.10 `src/db/index.ts` (DB client)

Drizzle client setup with `node-postgres` (`pg`). Reads `DATABASE_URL` from env. Exports `db` instance and a `withTransaction` helper that wraps operations in a transaction with `SET LOCAL` for RLS variables.

## 4. Zod Validation Schemas

### 4.1 `common.schema.ts`

```typescript
paginationParams    // page: z.coerce.number().int().min(1).default(1)
                    // limit: z.coerce.number().int().min(1).max(100).default(20)
uuidParam           // z.string().uuid()
signedUrlInput      // filename: z.string().min(1), contentType: z.enum(['image/jpeg', 'image/png', 'image/webp'])
```

### 4.2 `user.schema.ts`

```typescript
createUserInput     // name, email (z.string().email()), role (z.enum USER_ROLES),
                    // branch_id (uuid), team_id? (uuid), department?, position?
updateUserInput     // createUserInput.partial() — all fields optional
usersQueryParams    // extends paginationParams + branch_id?, team_id?, department?, is_active?, role?
```

### 4.3 `team.schema.ts`

```typescript
createTeamInput     // name (z.string().min(1).max(100)), branch_id (uuid), leader_id? (uuid)
updateTeamInput     // createTeamInput.partial()
```

### 4.4 `point.schema.ts`

Three separate submit schemas — not a discriminated union. Each enforces its own category's rules:

```typescript
submitBintangInput  // user_id (uuid), reason (z.string().min(1)),
                    // related_staff? (string), screenshot_url (z.string().url() — required)
                    // Note: points is NOT included — server hardcodes 1 for Bintang

submitPenaltiInput  // user_id, reason, kitta_component (z.enum(['K','I','T1','T2','A'])),
                    // points (z.number().int().min(1).max(10)),
                    // related_staff?, screenshot_url (required)

submitPoinAhaInput  // user_id, reason, points (z.number().int().min(1).max(10)),
                    // related_staff?, screenshot_url? (optional)

pointsQueryParams   // extends paginationParams + user_id?, category?, status?

approveInput        // (no body required)
rejectInput         // reason (z.string().min(1) — required for rejection)
revokeInput         // reason (z.string().min(1) — required)
```

### 4.5 `dispute.schema.ts`

```typescript
fileChallengeInput  // reason (z.string().min(1))
fileAppealInput     // reason (z.string().min(1))
resolveDisputeInput // status (z.enum(['upheld', 'overturned'])), resolution_note (z.string().min(1))
```

### 4.6 `reward.schema.ts`

```typescript
createRewardInput   // name (z.string().min(1).max(200)), description?, point_cost (z.number().int().min(1)),
                    // image_url?, branch_id? (uuid, nullable)
updateRewardInput   // createRewardInput.partial()
requestRedemptionInput // reward_id (uuid), notes?
rejectRedemptionInput  // reason (z.string().min(1))
redemptionsQueryParams // extends paginationParams + user_id?, status?
```

### 4.7 `social.schema.ts`

```typescript
createCommentInput  // entity_type (z.enum(['achievement','challenge','appeal'])),
                    // entity_id (uuid), body (z.string().min(1))
updateCommentInput  // body (z.string().min(1))
commentsQueryParams // entity_type (required), entity_id (uuid, required) — both required to scope query
notificationsQueryParams // extends paginationParams + is_read? (boolean)
```

### 4.8 `settings.schema.ts`

```typescript
updateSettingInput  // value (z.union([z.string(), z.number(), z.boolean()]))
                    // Settings store JSONB but values are simple scalars (impact numbers, flags)
createCategoryInput // code, default_name, description?, icon?, requires_screenshot (boolean)
updateCategoryInput // createCategoryInput.partial().omit({ code: true })
```

### 4.9 `leaderboard.schema.ts`

```typescript
leaderboardQueryParams // extends paginationParams + branch_id?, team_id?,
                       // type (z.enum(['bintang', 'poin_aha']) — required)
```

### 4.10 `dashboard.schema.ts`

```typescript
dashboardQueryParams   // branch_id? (uuid)
```

### 4.11 `admin.schema.ts`

```typescript
auditLogQueryParams    // extends paginationParams + branch_id?, actor_id?, action?, from? (date), to? (date), entity_type?
updateBranchInput      // name?, timezone?, locale? — all optional
```

## 5. TypeScript Types

### Pattern

Each types file exports:
1. **Inferred types** from Drizzle `$inferSelect` / `$inferInsert`
2. **Derived types** for API response shapes that join multiple tables

### 5.1 `common.ts`

```typescript
// API response envelope matching architecture Section 8
type ApiResponse<T> = {
  success: boolean
  data: T
  error: string | null
  meta?: PaginationMeta
}

type PaginationMeta = {
  total: number
  page: number
  limit: number
}

type PaginatedResponse<T> = ApiResponse<T[]> & { meta: PaginationMeta }
```

### 5.2 `user.ts`

```typescript
type Branch = typeof branches.$inferSelect
type NewBranch = typeof branches.$inferInsert
type SystemSetting = typeof systemSettings.$inferSelect

type User = typeof users.$inferSelect
type NewUser = typeof users.$inferInsert
type Team = typeof teams.$inferSelect
type NewTeam = typeof teams.$inferInsert

type UserWithTeam = User & { team: Team | null }
type TeamWithLeader = Team & { leader: User | null }
type TeamWithMembers = Team & { leader: User | null; members: User[] }
```

### 5.3 `point.ts`

```typescript
type AchievementPoint = typeof achievementPoints.$inferSelect
type NewAchievementPoint = typeof achievementPoints.$inferInsert
type PointCategory = typeof pointCategories.$inferSelect
type PointCategoryTranslation = typeof pointCategoryTranslations.$inferSelect

  category: PointCategory
  user: Pick<User, 'id' | 'name' | 'avatar_url'>
  submittedBy: Pick<User, 'id' | 'name' | 'role'>
  approvedBy: Pick<User, 'id' | 'name'> | null
  challenges: Challenge[]
  appeals: Appeal[]
  commentCount: number

// Leaderboard entry

  rank: number
  user: Pick<User, 'id' | 'name' | 'avatar_url' | 'department'>
  score: number  // bintang_count or computed poin_aha_balance

// Dashboard summary

  bintangCount: number
  poinAhaBalance: number

  pendingActions: number
}

type PointCategoryCode = 'BINTANG' | 'PENALTI' | 'POIN_AHA'

```

## 8. DB Client (`src/db/index.ts`)

Minimal Drizzle setup:

```typescript
// Reads DATABASE_URL from environment
// Creates node-postgres Pool
// Exports drizzle(pool, { schema }) as `db`
// Exports withTransaction(callback) that:
//   1. Begins transaction
//   2. SET LOCAL app.current_user_id, app.current_branch_id, app.current_role, app.current_team_id
//   3. Runs callback
//   4. Commits (or rolls back on error)
```

The `withTransaction` helper is critical for RLS — every API request must set session variables via `SET LOCAL` within a transaction scope.
