## 1. Overview

Translate the SQL DDL from `docs/architecture.md` Section 6 into a type-safe TypeScript domain layer using Drizzle ORM, Zod, and shared constants. Also update the architecture doc's monorepo structure and references to reflect the TanStack Start v1 migration.

### What this covers

- **Drizzle ORM schema** вЂ” 7 schema files grouped by bounded context
- **Zod validation schemas** вЂ” request input and query param validation for all API endpoints
- **TypeScript types** вЂ” inferred DB types + derived response types
- **Constants/enums** вЂ” roles, statuses, KITTA categories, notification types, settings keys
- **Architecture doc update** вЂ” Section 5 monorepo structure + scattered Next.js references

### What this does NOT cover

- Repository layer (`src/server/repositories/`) вЂ” designed with API routes
- Service layer (`src/server/services/`) вЂ” designed with business logic
- Migration files вЂ” generated by `drizzle-kit generate` after schema is written
- Seed data вЂ” separate task after migrations exist
- DB triggers/functions (updated_at, point_summary sync) вЂ” raw SQL in migration seed

### Bounded context grouping rationale

| Context | Tables | Why grouped |
|---------|--------|-------------|
| branches | branches, system_settings | Tenant config вЂ” rarely changes, referenced by everything |
| users | users, teams | Identity + org structure, circular FK (team.leader_id в†’ users) |
| points | achievement_points, point_categories, point_category_translations, point_summaries | Core transaction + its lookup tables and materialized view |
| disputes | challenges, appeals | Both are dispute mechanisms on achievement_points, separate auth rules |
| rewards | rewards, redemptions | Catalog + spend transactions |
| social | comments, notifications | User-facing communication, polymorphic patterns |
| audit | audit_logs, sheet_sync_jobs | Immutable records, admin-only |

## 3. Drizzle Schema Details

### 3.1 Common patterns

All schema files follow these conventions:

- **UUID primary keys** вЂ” `uuid('id').primaryKey().defaultRandom()`
- **Timestamps** вЂ” `timestamp('created_at', { withTimezone: true, mode: 'date' }).notNull().defaultNow()`
- **Foreign keys** вЂ” inline `.references(() => table.id)` for simple FKs
- **Enums** вЂ” `pgEnum` for PostgreSQL enums (`user_role`, `point_status`, `redemption_status`)
- **CHECK constraints** вЂ” Drizzle `check()` for value restrictions (kitta_component, point_categories.code)
- **Indexes** вЂ” `index('idx_name').on(table.column)` matching the architecture SQL indexes
- **Relations** вЂ” defined via `relations()` for Drizzle's relational query API

### 3.2 `branches.ts`

```typescript
// Tables: branches, systemSettings
// Enums: none
// Relations: systemSettings.updatedBy в†’ users (defined in users.ts to avoid circular import)
```

`branches` вЂ” id, code (varchar 10, unique), name (varchar 100), timezone (varchar 50, default 'Asia/Jakarta'), locale (varchar 10, default 'id'), created_at, updated_at.

`systemSettings` вЂ” key (varchar 100, PK), value (jsonb), description (text), updated_by (uuid FK в†’ users), updated_at. No `created_at` вЂ” settings are seeded, not user-created.

### 3.3 `users.ts`

```typescript
// Tables: users, teams
// Enums: userRoleEnum
// Relations: users в†” teams (bidirectional), teams.leader в†’ users
```

`userRoleEnum` вЂ” `pgEnum('user_role', ['admin', 'hr', 'leader', 'employee'])`

`teams` вЂ” id, branch_id FK, name (varchar 100), leader_id (uuid, nullable вЂ” circular FK resolved via separate `.references()`), created_at, updated_at. Indexes: branch_id, leader_id.

`users` вЂ” id, branch_id FK, team_id FK (nullable), email (varchar 255, unique), name (varchar 255), employee_id (varchar 50, nullable), role (userRoleEnum, default 'employee'), department (varchar 100, nullable), position (varchar 100, nullable), avatar_url (text, nullable), locale_pref (varchar 10, nullable), must_change_password (boolean, default true), is_active (boolean, default true), archived_at (timestamp, nullable), created_at, updated_at. Indexes: branch_id, team_id, email.

Relations:
- `users.branch` в†’ one branches
- `users.team` в†’ one teams (nullable)
- `teams.branch` в†’ one branches
- `teams.leader` в†’ one users (nullable)
- `teams.members` в†’ many users

### 3.4 `points.ts`

```typescript
// Tables: pointCategories, pointCategoryTranslations, achievementPoints, pointSummaries
// Enums: pointStatusEnum
// Relations: achievementPoints в†’ user, submittedBy, approvedBy, revokedBy, category, branch
```

`pointStatusEnum` вЂ” `pgEnum('point_status', ['pending', 'active', 'challenged', 'frozen', 'revoked', 'rejected'])`

`pointCategories` вЂ” id, code (varchar 50, unique, CHECK IN ('BINTANG', 'PENALTI', 'POIN_AHA')), default_name (varchar 100), description (text, nullable), icon (varchar 50, nullable), requires_screenshot (boolean, default true), is_active (boolean, default true), created_at.

`pointCategoryTranslations` вЂ” id, category_id FK, locale (varchar 10), name (varchar 100), description (text, nullable). UNIQUE(category_id, locale).

`achievementPoints` вЂ” id, branch_id FK, user_id FK, category_id FK, points (integer, CHECK > 0), reason (text), related_staff (text, nullable), screenshot_url (text, nullable), kitta_component (varchar 2, nullable, CHECK IN ('K', 'I', 'T1', 'T2', 'A') OR NULL), status (pointStatusEnum, default 'pending'), submitted_by FK, approved_by FK (nullable), approved_at (nullable), revoked_by FK (nullable), revoked_at (nullable), revoke_reason (text, nullable), metadata (jsonb, default '{}'), created_at, updated_at. Indexes: branch_id, user_id, status, category_id, submitted_by, created_at.

`pointSummaries` вЂ” id, branch_id FK, user_id FK (UNIQUE), bintang_count (integer, default 0), penalti_points_sum (integer, default 0), direct_poin_aha (integer, default 0), redeemed_total (integer, default 0), updated_at. Indexes: (branch_id, bintang_count DESC), (branch_id, direct_poin_aha DESC).

### 3.5 `disputes.ts`

```typescript
// Tables: challenges, appeals
// Enums: none (status is varchar CHECK)
// Relations: в†’ achievementPoints, challenger/appellant в†’ users, resolvedBy в†’ users
```

Both tables share the same shape: id, branch_id FK, achievement_id FK, [challenger_id | appellant_id] FK, reason (text), status (varchar 20, CHECK IN ('open', 'upheld', 'overturned'), default 'open'), resolved_by FK (nullable), resolved_at (nullable), resolution_note (text, nullable), created_at, updated_at. Indexes: achievement_id, branch_id.

### 3.6 `rewards.ts`

```typescript
// Tables: rewards, redemptions
// Enums: redemptionStatusEnum
```

`redemptionStatusEnum` вЂ” `pgEnum('redemption_status', ['pending', 'approved', 'rejected'])`

`rewards` вЂ” id, branch_id FK (nullable вЂ” NULL = global), name (varchar 200), description (text, nullable), point_cost (integer, CHECK > 0), image_url (text, nullable), is_active (boolean, default true), created_at, updated_at. Index: branch_id.

`redemptions` вЂ” id, branch_id FK, user_id FK, reward_id FK, points_spent (integer, CHECK > 0), notes (text, nullable), status (redemptionStatusEnum, default 'pending'), approved_by FK (nullable), approved_at (nullable), rejection_reason (text, nullable), created_at, updated_at. Indexes: user_id, branch_id.

### 3.7 `social.ts`

```typescript
// Tables: comments, notifications
// Relations: comments.author в†’ users, notifications.user в†’ users
```

`comments` вЂ” id, branch_id FK, entity_type (varchar 30, CHECK IN ('achievement', 'challenge', 'appeal')), entity_id (uuid вЂ” polymorphic, no DB FK), author_id FK, body (text), created_at, updated_at. Index: (entity_type, entity_id), branch_id.

`notifications` вЂ” id, branch_id FK, user_id FK, type (varchar 50), title (text), body (text, nullable), entity_type (varchar 30, nullable), entity_id (uuid, nullable), is_read (boolean, default false), read_at (timestamp, nullable), created_at. Indexes: (user_id, is_read), (read_at WHERE is_read = true), branch_id.

### 3.8 `audit.ts`

```typescript
// Tables: auditLogs, sheetSyncJobs
// Relations: auditLogs.actor в†’ users
```

`auditLogs` вЂ” id, branch_id FK, actor_id FK, action (varchar 50), entity_type (varchar 50), entity_id (uuid), old_value (jsonb, nullable), new_value (jsonb, nullable), ip_address (custom inet type via `customType`), created_at. Indexes: branch_id, actor_id, (entity_type, entity_id). No updated_at вЂ” immutable.

`sheetSyncJobs` вЂ” id, branch_id FK, direction (varchar 10, CHECK IN ('import', 'export')), sheet_id (varchar 255), sheet_name (varchar 100, nullable), status (varchar 20, default 'pending'), rows_processed (integer, default 0), rows_failed (integer, default 0), error_log (jsonb, nullable), started_by FK (nullable), started_at (nullable), completed_at (nullable), created_at.

### 3.9 `index.ts` (barrel)

Re-exports all tables, enums, and relations from all schema files. Single import point:
```typescript
import { users, teams, achievementPoints, ... } from '@/db/schema'
```

### 3.10 `src/db/index.ts` (DB client)

Drizzle client setup with `node-postgres` (`pg`). Reads `DATABASE_URL` from env. Exports `db` instance and a `withTransaction` helper that wraps operations in a transaction with `SET LOCAL` for RLS variables.

## 4. Zod Validation Schemas

### 4.1 `common.schema.ts`

```typescript
paginationParams    // page: z.coerce.number().int().min(1).default(1)
                    // limit: z.coerce.number().int().min(1).max(100).default(20)
uuidParam           // z.string().uuid()
signedUrlInput      // filename: z.string().min(1), contentType: z.enum(['image/jpeg', 'image/png', 'image/webp'])
```

### 4.2 `user.schema.ts`

```typescript
createUserInput     // name, email (z.string().email()), role (z.enum USER_ROLES),
                    // branch_id (uuid), team_id? (uuid), department?, position?
updateUserInput     // createUserInput.partial() вЂ” all fields optional
usersQueryParams    // extends paginationParams + branch_id?, team_id?, department?, is_active?, role?
```

### 4.3 `point.schema.ts`

Three separate submit schemas вЂ” not a discriminated union. Each enforces its own category's rules:

```typescript
submitBintangInput  // user_id (uuid), reason (z.string().min(1)),
                    // related_staff? (string), screenshot_url (z.string().url() вЂ” required)

submitPenaltiInput  // user_id, reason, kitta_component (z.enum(['K','I','T1','T2','A'])),
                    // points (z.number().int().min(1).max(10)),
                    // related_staff?, screenshot_url (required)

submitPoinAhaInput  // user_id, reason, points (z.number().int().min(1).max(10)),
                    // related_staff?, screenshot_url? (optional)

pointsQueryParams   // extends paginationParams + user_id?, category?, status?

approveRejectInput  // reason? (string, optional вЂ” used for rejection reason)
revokeInput         // reason (z.string().min(1) вЂ” required)
```

### 4.4 `dispute.schema.ts`

```typescript
fileChallengeInput  // reason (z.string().min(1))
fileAppealInput     // reason (z.string().min(1))
resolveDisputeInput // status (z.enum(['upheld', 'overturned'])), resolution_note (z.string().min(1))
```

### 4.5 `reward.schema.ts`

```typescript
createRewardInput   // name (varchar 200), description?, point_cost (z.number().int().min(1)),
                    // image_url?, branch_id? (uuid, nullable)
updateRewardInput   // createRewardInput.partial()
requestRedemptionInput // reward_id (uuid), notes?
rejectRedemptionInput  // reason (z.string().min(1))
```

### 4.6 `social.schema.ts`

```typescript
createCommentInput  // entity_type (z.enum(['achievement','challenge','appeal'])),
                    // entity_id (uuid), body (z.string().min(1))
updateCommentInput  // body (z.string().min(1))
```

### 4.7 `settings.schema.ts`

```typescript
updateSettingInput  // value (z.union([z.string(), z.number(), z.boolean()]))
                    // Settings store JSONB but values are simple scalars (impact numbers, flags)
```

## 5. TypeScript Types

### Pattern

Each types file exports:
1. **Inferred types** from Drizzle `$inferSelect` / `$inferInsert`
2. **Derived types** for API response shapes that join multiple tables

### 5.1 `user.ts`

```typescript
type User = typeof users.$inferSelect
type NewUser = typeof users.$inferInsert
type Team = typeof teams.$inferSelect
type NewTeam = typeof teams.$inferInsert

type UserWithTeam = User & { team: Team | null }
type TeamWithLeader = Team & { leader: User | null }
type TeamWithMembers = Team & { leader: User | null; members: User[] }
```

### 5.2 `point.ts`

```typescript
type AchievementPoint = typeof achievementPoints.$inferSelect
type NewAchievementPoint = typeof achievementPoints.$inferInsert
type PointCategory = typeof pointCategories.$inferSelect
type PointSummary = typeof pointSummaries.$inferSelect

// API response shape for point detail page
type AchievementPointDetail = AchievementPoint & {
  category: PointCategory
  user: Pick<User, 'id' | 'name' | 'avatar_url'>
  submittedBy: Pick<User, 'id' | 'name' | 'role'>
  approvedBy: Pick<User, 'id' | 'name'> | null
  challenges: Challenge[]
  appeals: Appeal[]
  commentCount: number
}

// Leaderboard entry
type LeaderboardEntry = {
  rank: number
  user: Pick<User, 'id' | 'name' | 'avatar_url' | 'department'>
  score: number  // bintang_count or computed poin_aha_balance
}

type PointCategoryCode = 'BINTANG' | 'PENALTI' | 'POIN_AHA'
type PointStatus = 'pending' | 'active' | 'challenged' | 'frozen' | 'revoked' | 'rejected'
```

### 5.3 `dispute.ts`

```typescript
type Challenge = typeof challenges.$inferSelect
type Appeal = typeof appeals.$inferSelect
type DisputeStatus = 'open' | 'upheld' | 'overturned'

type ChallengeDetail = Challenge & {
  challenger: Pick<User, 'id' | 'name' | 'role'>
  resolvedBy: Pick<User, 'id' | 'name'> | null
}

type AppealDetail = Appeal & {
  appellant: Pick<User, 'id' | 'name'>
  resolvedBy: Pick<User, 'id' | 'name'> | null
}
```

### 5.4 `reward.ts`

```typescript
type Reward = typeof rewards.$inferSelect
type Redemption = typeof redemptions.$inferSelect

type RedemptionWithReward = Redemption & {
  reward: Pick<Reward, 'id' | 'name' | 'image_url' | 'point_cost'>
  approvedBy: Pick<User, 'id' | 'name'> | null
}
```

### 5.5 `social.ts`

```typescript
type Comment = typeof comments.$inferSelect
type Notification = typeof notifications.$inferSelect

type CommentWithAuthor = Comment & {
  author: Pick<User, 'id' | 'name' | 'avatar_url' | 'role'>
}

  | 'challenge_resolved' | 'appeal_resolved' | 'redemption_approved'
  | 'redemption_rejected' | 'self_submit_approved' | 'self_submit_rejected'
```

### 5.6 `audit.ts`

```typescript
type AuditLog = typeof auditLogs.$inferSelect

  actor: Pick<User, 'id' | 'name' | 'email' | 'role'>
}
```
